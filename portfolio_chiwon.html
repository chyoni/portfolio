TROUBLE 01

문제

TROUBLE 02
동시 수동 동기화 요청 처리
문제
수동 동기화는 여러 사용자가 동시에 같은 티켓에 대해 동기화 버튼을 누를 수 있었습니다. 사용자 A와 B가 동시에 티켓 #123을 수동 동기화하면(심지어 티켓을 각각 수정하고 난 후), A의 요청과 B의 요청이 각각 소스를 읽고 타겟에 쓰기 시작하면서 마지막에 쓴 요청이 이기거나, 최악의 경우 데이터가 섞이는 현상이 발생했습니다. 또한 같은 커넥션에 대해 동시에 여러 동기화가 돌면 외부 API의 Rate Limit에 걸렸습니다.
해결방법
두 가지 레벨에서 제어했습니다.

① 티켓 레벨 락 — 재시도 + 대기
SyncLock 테이블에 connection_id + ticket_id를 유니크 키로 두고, 동기화 시작 시 INSERT, 완료 시 DELETE하는 방식입니다. 락 습득 실패 시 바로 에러를 던지지 않고 일정 시간 대기 후 재시도합니다. 단, finally에서 무조건 releaseLock() 을 호출하면 락 습득에 실패한 경우에도 남의 락을 삭제 시도하는 문제가 생기므로, acquired 플래그로 "내가 잡은 경우에만 해제"를 보장했습니다.
public void syncWithRetry(Long connectionId, String ticketId) {
    int maxAttempts = 5;

    for (int attempt = 1; attempt <= maxAttempts; attempt++) {
        boolean acquired = false;
        try {
            syncLockRepository.acquireLock(connectionId, ticketId);
            acquired = true;

            doSync(connectionId, ticketId);
            return;

        } catch (DuplicateKeyException e) {
            if (attempt == maxAttempts) {
                // 끝까지 락을 못 잡으면 그때 에러
                throw new SyncAlreadyInProgressException(ticketId);
            }
            // 500ms → 1s → 1.5s → 2s 점진적 대기 후 재시도
            Thread.sleep(500L * attempt);

        } finally {
            // 내가 실제로 락을 잡은 경우에만 해제
            if (acquired) {
                syncLockRepository.releaseLock(connectionId, ticketId);
            }
        }
    }
}
② 커넥션 레벨 스로틀링
커넥션별로 동시 실행 가능한 최대 스레드 수를 제한하는 Semaphore를 두었습니다.
private final Map<Long, Semaphore> connectionSemaphores = new ConcurrentHashMap<>();

public void syncWithThrottle(Long connectionId, Runnable syncTask) {
    Semaphore semaphore = connectionSemaphores
        .computeIfAbsent(connectionId, id -> new Semaphore(3)); // 최대 3개 동시
    semaphore.acquire();
    try {
        syncTask.run();
    } finally {
        semaphore.release();
    }
}
결과
커넥션 별로 세마포어를 설정해 한 커넥션 당 최대 동시 3개 요청을 허용하고, 그 요청 중에서 같은 티켓이 중복 동기화되는 현상이 제거됐습니다. 외부 API Rate Limit 초과로 인한 장애도 방지할 수 있었습니다.
TROUBLE 03
불필요한 재동기화 방지 + 신규 티켓 중복 생성 방지
문제
[업데이트 케이스]
기준점이 없으면 스케줄러 실행마다 등록된 티켓 전체를 동기화 시도합니다. 실제로 오늘 변경된 티켓이 3개뿐인데 10만 번 API를 호출하게 되어 Rate Limit 초과와 성능 저하가 발생했습니다.
[신규 생성 케이스]
동기화 흐름은 ① 소스 읽기 → ② 필드 매핑 → ③ 타겟에 티켓 생성(외부 API) → ④ DB에 매핑 기록 순서입니다. ③이 성공하고 ④가 실패하면, 다음 스케줄에서 매핑 정보가 없어 소스 티켓을 신규로 판단하고 타겟에 티켓을 또 생성합니다. ③이 외부 API 호출이기 때문에 트랜잭션으로 묶는 것은 불가능합니다.
해결방법
[업데이트 케이스] SyncHistory — 동기화 필요 여부 판단 기준점
SyncHistory의 역할은 정합성 보장이나 멱등성이 아닙니다. "이 티켓을 지금 동기화할 필요가 있는가?"를 판단하는 필터입니다. source_updated_at > last_synced_at 인 티켓만 동기화 대상으로 추려냅니다. ③ 성공 + ④ 실패로 재동기화가 발생해도, 업데이트는 덮어쓰기 구조라 원래부터 멱등하기 때문에 데이터 이상은 없습니다. 
connection_id     -- 어느 커넥션
source_ticket_id  -- 소스 티켓 ID
last_synced_at    -- 마지막 성공 시각
source_updated_at -- 소스 티켓의 마지막 수정 시각
sync_status       -- SUCCESS / FAILED / IN_PROGRESS

-- 아래 조건을 만족하는 티켓만 동기화 대상으로 추출
-- source_updated_at > last_synced_at
[신규 생성 케이스] TicketMapping 테이블 + 타겟 롤백
소스-타겟 티켓 ID 매핑을 별도 테이블에 저장합니다. 동기화 전 매핑이 존재하면 생성을 건너뛰고 업데이트만 수행합니다. 타겟 생성 후 DB 저장이 실패하면 타겟 티켓을 직접 삭제해 롤백합니다. 
public void syncNewTicket(String sourceTicketId) {

    // 이미 생성된 적 있으면 업데이트만
    Optional<TicketMapping> existing =
        ticketMappingRepository.findBySourceTicketId(sourceTicketId);

    if (existing.isPresent()) {
        updateTicket(existing.get().targetTicketId(), sourceTicketId);
        return;
    }

    String targetTicketId = null;
    try {
        // 타겟에 티켓 생성
        targetTicketId = targetClient.createTicket(sourceTicketId);

        // 매핑 + 히스토리 저장
        ticketMappingRepository.save(
            TicketMapping.of(sourceTicketId, targetTicketId)
        );
        syncHistoryRepository.save(
            SyncHistory.ofNewTicket(sourceTicketId, targetTicketId)
        );

    } catch (Exception e) {
        // DB 저장 실패 시 타겟 티켓 롤백
        if (targetTicketId != null) {
            targetClient.deleteTicket(targetTicketId);
        }
        throw e;
    }
}
결과
두 테이블이 각자 다른 역할로 문제를 해결합니다.

SyncHistory — 변경된 티켓만 골라내는 필터. 10만 건 중 오늘 변경된 3건만 API를 호출하게 됩니다.

TicketMapping — 소스 티켓에 대응하는 타겟 티켓이 이미 존재하는지 확인. 중복 생성을 원천 차단합니다.
TROUBLE 04
외부 API 변경 & 버전 대응
문제
JIRA Server 8.x와 10.x는 REST API 엔드포인트 자체가 다르고, Codebeamer도 버전마다 스펙이 달라 고객사마다 다른 구현체가 필요했습니다. 시스템 타입(JIRA_SERVER, CODEBEAMER)만으로 빈을 구분하면 버전 정보가 누락되고, 소스와 타겟이 각각 다른 시스템일 때 클라이언트를 두 벌 꺼내야 한다는 문제도 있었습니다.
해결방법
추상화 인터페이스 TicketSystemClient를 두고 버전별 구현체를 분리한 뒤, TicketSystemClientFactory가 systemType + version 조합으로 적절한 구현체를 반환하도록 설계했습니다. 커넥션 테이블에 소스/타겟 각각의 시스템 타입과 버전을 저장해두고, 동기화 시작 시 Factory에서 소스·타겟 클라이언트를 각각 꺼내 사용합니다. 
// 추상화 인터페이스
interface TicketSystemClient {
    List<Ticket> fetchTickets(String projectKey, LocalDateTime since);
    void updateTicket(String ticketId, Map<String, Object> fields);
}

// 버전별 구현체 분리
@Component class JiraServer8Client  implements TicketSystemClient { ... }
@Component class JiraServer10Client implements TicketSystemClient { ... }
@Component class CodebeamerClient   implements TicketSystemClient { ... }
// Factory — systemType + version 조합 로직 캡슐화
@Component
class TicketSystemClientFactory {

    private final JiraServer8Client  jiraServer8Client;
    private final JiraServer10Client jiraServer10Client;
    private final CodebeamerClient   codebeamerClient;

    public TicketSystemClientFactory(
        JiraServer8Client  jiraServer8Client,
        JiraServer10Client jiraServer10Client,
        CodebeamerClient   codebeamerClient
    ) {
        this.jiraServer8Client  = jiraServer8Client;
        this.jiraServer10Client = jiraServer10Client;
        this.codebeamerClient   = codebeamerClient;
    }

    public TicketSystemClient resolve(String systemType, String version) {
        return switch (systemType) {
            case "JIRA_SERVER" -> resolveJiraServer(version);
            case "CODEBEAMER"  -> codebeamerClient;
            default -> throw new UnsupportedSystemException(systemType);
        };
    }

    private TicketSystemClient resolveJiraServer(String version) {
        int major = Integer.parseInt(version.split("\\.")[0]);
        return major >= 10 ? jiraServer10Client : jiraServer8Client;
    }
}
// SyncService — 소스/타겟 클라이언트 각각 꺼내기
public void sync(Connection connection) {
    TicketSystemClient sourceClient = clientFactory.resolve(

    );
    TicketSystemClient targetClient = clientFactory.resolve(
        connection.targetSystemType(), connection.targetVersion()
    );

    List<Ticket> tickets = sourceClient.fetchTickets(...);
    targetClient.updateTicket(...);
}

TROUBLE 05
last_synced_at 기준 시간 오염
문제
가장 흔한 실수는 last_synced_at에 서버 현재 시간(LocalDateTime.now())을 저장하는 것입니다. 소스 티켓을 읽은 시점(T1)과 DB에 last_synced_at을 저장하는 시점(T3) 사이에 소스가 변경되면 — T3가 소스의 새로운 updatedAt(T2)보다 크기 때문에 — 그 변경을 영구적으로 놓치게 됩니다. 
// 잘못된 방식 — 서버 현재 시간 저장
// (이렇게 하면 T1~T3 사이 소스 변경을 영구 누락)

// T1: 소스 읽기 → source_updated_at = T1
// T2: (이 사이 소스 티켓 수정됨) → source_updated_at = T2
// T3: last_synced_at = LocalDateTime.now() 저장
// 결과: T3 > T2 → 다음 스케줄에서 변경 감지 안 됨
해결방법
last_synced_at에는 반드시 소스 시스템이 알려준 updatedAt 값을 그대로 저장해야 합니다. 동기화 시작 시 소스의 updatedAt을 미리 기록해두고, 성공 시에만 그 값으로 확정짓는 방식을 사용했습니다. 
// 올바른 방식 — 소스의 updatedAt 기준으로 저장

// 동기화 시작 시 — 소스 기준 시점을 미리 기록
history.markInProgress(sourceTicket.getUpdatedAt());

// 동기화 성공 시 — 소스 기준으로 확정
history.confirmSync();

// T1: 소스 읽기 → inProgressSince = T1 기록
// T2: (이 사이 소스 티켓 수정됨) → source_updated_at = T2
// 성공: last_synced_at = T1 저장
// 결과: T2 > T1 → 다음 스케줄에서 변경 정상 감지
결과
읽는 시점과 저장 시점 사이에 소스가 변경되더라도, last_synced_at은 "우리가 마지막으로 읽은 버전의 시간"으로 정확히 남아 다음 스케줄에서 변경을 놓치지 않게 됐습니다.
TROUBLE 06
타겟 직접 수정 시 Lost Update
문제
소스가 Master라는 전제가 있더라도, 실제 운영에서는 타겟 JIRA, Codebeamer에서 담당자가 직접 티켓을 수정하는 일이 생겼습니다. 그리고 다음 동기화 사이클에 소스 값이 타겟을 덮어쓰면서 타겟의 수동 수정 내용이 조용히 사라졌습니다. 신고가 들어오기 전까지는 아무도 모르는 경우가 많았습니다.
해결방법
① 쓰기 전 타겟 현재 값과 비교 (Conflict Detection)
동기화 시 타겟에 바로 쓰지 않고, 먼저 타겟의 현재 값을 읽어서 "시스템이 마지막으로 쓴 값"과 비교했습니다. 
String lastWrittenValue = syncHistory.getLastWrittenValue(fieldName);
String currentTargetValue = targetClient.getFieldValue(ticketId, fieldName);
String newSourceValue = sourceTicket.getFieldValue(fieldName);

if (!currentTargetValue.equals(lastWrittenValue)) {
    // 타겟에서 직접 수정이 있었음 → Conflict 발생
    conflictLogService.record(ticketId, fieldName,
        lastWrittenValue, currentTargetValue, newSourceValue);
    // 정책에 따라: 소스 우선 덮어쓰기 or 스킵 or 알림

TROUBLE 09
양방향 동기화 무한 루프
문제
JIRA → Codebeamer, Codebeamer → JIRA를 양방향으로 설정하면 다음과 같은 루프가 발생했습니다.
① JIRA 티켓 변경 감지 → ② Codebeamer에 쓰기 (Codebeamer updatedAt 갱신) → ③ Codebeamer 변경 감지 (우리가 쓴 건데 변경으로 인식) → ④ JIRA에 다시 쓰기 → ① 로 돌아감.
로그엔 계속 "동기화 성공"만 찍히니까 처음엔 정상처럼 보이다가 외부 API 호출량이 폭발하면서 Rate Limit에 걸리거나 성능 저하의 주 원인이 됐습니다.
해결방법
"시스템이 쓴 변경"과 "사람이 만든 변경"을 구분하기 위해 SyncWriteLog 테이블을 별도로 두는 방식을 채택했습니다. 타겟에 쓸 때 반드시 로그를 기록하고, 반대 방향 스케줄러가 변경을 감지했을 때 해당 티켓이 최근에 우리 시스템이 쓴 것인지 이 테이블에서 확인한 뒤 스킵 여부를 결정합니다.

// 반대 방향 스케줄러 — 변경 감지 시 우리가 쓴 것인지 먼저 확인
public boolean isRecentlySyncWritten(Long connectionId,
                                      String targetTicketId,
                                      LocalDateTime ticketUpdatedAt) {
    return syncWriteLogRepository
        .findMostRecentWrite(connectionId, targetTicketId)
        .map(log -> {
            // 우리가 쓴 시각 이후 updatedAt이 더 최신이면 사람이 다시 수정한 것
            // 5초 버퍼 — 타겟 시스템의 updatedAt 갱신 지연을 감안한 값
            return !ticketUpdatedAt.isAfter(log.writtenAt().plusSeconds(5));
        })
        .orElse(false);
}

private final Map<Long, ExecutorService> connectionExecutors = new ConcurrentHashMap<>();

private ExecutorService resolveExecutor(Long connectionId) {

        id -> Executors.newFixedThreadPool(5));
}

public void processBatch(Long connectionId, List<Ticket> tickets) {

        .map(ticket -> CompletableFuture.runAsync(

        .toList();

CompletableFuture.supplyAsync()로 AIP 암호화를 병렬 실행하는 전용 스레드 풀(bulkDownloadAipExecutor, corePoolSize=5, maxPoolSize=10)을 설계했습니다. 암호화 완료 후 ZipOutputStream으로 디스크에 ZIP을 먼저 기록하고, StreamingResponseBody로 전송하는 방식을 채택했습니다. 스트리밍 ZIP 방식은 Content-Length를 사전에 알 수 없어 프론트엔드 진행률 표시가 불가능하므로, 디스크에 ZIP을 완성한 뒤 Files.size()로 정확한 Content-Length를 제공하는 구조를 선택했습니다. 일부 파일의 AIP 실패 시 성공한 파일만으로 ZIP을 구성하고 실패 정보를 X-Bulk-Download-Failed-Files 응답 헤더로 전달하는 부분 실패 허용 전략도 적용했습니다. 기존 단건 다운로드의 AIP 처리 로직은 DownloadFilePreparationHelper로 추출하여 단건·일괄 다운로드가 동일 코드를 공유하도록 구조를 개선했습니다.

TROUBLE 01
@Async self-invocation — 비동기가 동기로 동작하는 버그
문제
파일 병합·NAS 저장·감사 로그 등 후속 작업을 @Async로 비동기 분리했으나, 같은 클래스의 completeUpload()에서 processCompletionAsync()를 직접 호출하면 여전히 동기로 동작했습니다. 사용자는 청크 전송 완료 후에도 후속 처리가 끝날 때까지 응답을 받지 못했습니다.
해결방법
Spring의 @Async / @Transactional은 프록시 기반입니다. 같은 클래스 내 메서드 호출(this.xxx())은 프록시가 아닌 실제 객체를 호출하기 때문에 AOP가 무효화됩니다. 비동기 처리 메서드를 별도 UploadCompletionService로 분리하여 Spring이 관리하는 프록시 빈을 주입받아 호출하도록 리팩토링했습니다. 
// Before — self-invocation: 프록시 우회, @Async 무효
@Service
class UploadService {

    public void completeUpload(Long taskId) {
        validateAllChunksUploaded(taskId);
        processCompletionAsync(taskId); // this.xxx() → 프록시 거치지 않음
    }

    @Async
    public void processCompletionAsync(Long taskId) { ... }
}
// After — 별도 빈으로 분리, 프록시 통해 호출
@Service
class UploadService {

    private final UploadCompletionService completionService;

    public void completeUpload(Long taskId) {
        validateAllChunksUploaded(taskId);
        completionService.processAsync(taskId); // 프록시 빈 호출 → @Async 정상 동작
    }
}

@Service
class UploadCompletionService {

    @Async
    public void processAsync(Long taskId) {
        mergeChunks(taskId);
        saveToNas(taskId);
        recordAuditLog(taskId);
    }
}
결과
사용자는 청크 전송 완료 직후 응답을 받고 화면을 이탈할 수 있게 됐습니다. 후속 처리(병합·NAS 저장·감사 로그)는 백그라운드에서 독립적으로 진행되어 UX가 개선됐습니다.
TROUBLE 02
취소 버튼 눌러도 청크가 계속 전송되는 버그
문제
취소 버튼 클릭 시 ChunkManager.abort()가 호출되어 AbortController로 진행 중인 fetch를 중단했음에도, 네트워크 탭에서 남은 청크 업로드 요청이 계속 발생했습니다. AbortController는 "현재 진행 중인 네트워크 요청"만 취소합니다. 워커 루프(while chunks.length > 0)는 취소 여부를 전혀 확인하지 않아, AbortError로 현재 청크가 종료돼도 루프가 다음 청크를 계속 shift()하여 새 요청을 계속 보냈습니다.
해결방법
"I/O 취소"와 "작업 취소"는 다르다는 것을 파악하고, ChunkManager에 도메인 레벨 취소 플래그(cancelled)를 추가했습니다. 워커 루프 조건과 uploadSingleChunk 진입 시점 두 곳 모두에서 isCancelled()를 확인하도록 수정했습니다. 
// Before — AbortController만으로 취소 시도, 루프는 계속 돌아감
class ChunkManager {
    controller = new AbortController();

    abort() {
        this.controller.abort(); // 현재 요청만 중단
    }

    async startWorker() {
        while (this.chunks.length > 0) {        // 취소 여부 확인 없음
            const chunk = this.chunks.shift();
            await this.uploadSingleChunk(chunk); // AbortError → 다음 루프 계속 진행
        }
    }
}
// After — 도메인 레벨 취소 플래그 추가
class ChunkManager {
    controller = new AbortController();
    cancelled  = false;

    abort() {
        this.cancelled = true;  // 작업 취소 플래그
        this.controller.abort(); // 현재 I/O 취소
    }

    isCancelled() { return this.cancelled; }

    async startWorker() {
        while (this.chunks.length > 0 && !this.isCancelled()) { // 루프 진입 전 확인
            const chunk = this.chunks.shift();
            if (this.isCancelled()) break;          // 청크 처리 직전 재확인
            await this.uploadSingleChunk(chunk);
        }
    }
}
결과
취소 버튼 클릭 즉시 워커 루프가 중단되어 불필요한 네트워크 요청이 발생하지 않습니다. JS 싱글 스레드 특성상 여러 워커의 chunks.shift() 동시 접근은 중복 없이 안전하며, 취소 플래그로 "작업 취소" 시맨틱을 명확히 분리했습니다.
TROUBLE 03
대용량 파일 다운로드 OOM — StreamingResponseBody 도입
문제
다운로드 시 Files.readAllBytes()로 파일 전체를 byte[]로 읽어 JVM 힙에 올린 후 응답했습니다. 2GB 이상 파일에서는 int 범위 초과 및 OOM 위험이 존재했습니다.
해결방법
FileSystemResource 방식 대신 StreamingResponseBody를 도입했습니다. AIP 암복호화 후 생성되는 임시 파일을 InputStream으로 읽어 OutputStream으로 즉시 흘려보내고(transferTo), finally 블록에서 임시 파일을 삭제하는 FileDownloadStreamer 컴포넌트를 별도 설계했습니다. 
// Before — 전체를 힙에 올림, 2GB에서 OOM
public ResponseEntity<byte[]> download(Long fileId) {
    Path   decrypted = aipService.decrypt(fileId);
    byte[] bytes     = Files.readAllBytes(decrypted); // 2GB → OOM
    return ResponseEntity.ok(bytes);
}
// After — InputStream → OutputStream 스트리밍, 힙 사용량 버퍼 크기 수준 유지
@Component
class FileDownloadStreamer {

    public StreamingResponseBody stream(Path decryptedFile) {
        return outputStream -> {
            try (InputStream in = Files.newInputStream(decryptedFile)) {
                in.transferTo(outputStream); // 청크 단위로 흘려보냄
            } finally {
                Files.deleteIfExists(decryptedFile); // 임시 파일 정리
            }
        };
    }
}

// 컨트롤러 — 컨트롤러·서비스·스트리머 각 책임 분리
public ResponseEntity<StreamingResponseBody> download(Long fileId) {
    FileMetadata         meta      = fileService.findMetadata(fileId);
    Path                  decrypted = aipService.decrypt(fileId);
    StreamingResponseBody body      = streamer.stream(decrypted);

    return ResponseEntity.ok()
        .header(HttpHeaders.CONTENT_DISPOSITION,
                "attachment; filename=\"" + meta.originalFileName() + "\"")
        .contentType(MediaType.APPLICATION_OCTET_STREAM)
        .body(body);
}
결과
JVM 힙 사용량을 버퍼 크기(수 KB) 수준으로 유지하며 2GB 이상 파일도 안정적으로 다운로드할 수 있게 됐습니다. 컨트롤러·서비스·스트리머 각 책임이 명확히 분리되어 유지보수성도 향상됐습니다.
TROUBLE 04
다운로드 중 이탈 후 재시도 시 파일 손상 버그
문제
사용자가 다운로드 중 화면을 이탈했다가 동일 파일을 다시 다운로드 시도하면 파일을 받지 못하는 현상이 발생했습니다. 임시 파일 생성 시 원본 파일명을 그대로 사용했기 때문에, 이전 다운로드가 중단되어 훼손된 임시 파일이 NAS 공유 경로에 남아 있는 상태에서 동일 파일명으로 새 요청이 들어오면 훼손된 파일을 그대로 사용했습니다.
해결방법
임시 파일 생성 시 파일명을 UUID 난수값으로 변경했습니다. "선 삭제 후 재생성" 방안도 고려했으나, 삭제 명분이 불명확하고 응답 시간을 늘리는 부작용이 있어 채택하지 않았습니다. 매 요청마다 고유한 임시 파일이 생성되므로 이전 중단 잔여 파일의 영향을 받지 않습니다. 
// Before — 원본 파일명 그대로 사용, 중단 잔여 파일과 충돌
public Path decrypt(Long fileId) {
    FileMetadata meta     = fileService.findMetadata(fileId);
    Path         tempFile = tempDir.resolve(meta.originalFileName()); // 충돌 위험
    aipClient.decryptTo(meta.nasPath(), tempFile);
    return tempFile;
}
// After — UUID 난수값으로 임시 파일명 생성, 요청마다 고유
public Path decrypt(Long fileId) {
    FileMetadata meta     = fileService.findMetadata(fileId);
    String       tempName = UUID.randomUUID() + "_" + meta.originalFileName();
    Path         tempFile = tempDir.resolve(tempName); // 매 요청마다 고유한 경로
    aipClient.decryptTo(meta.nasPath(), tempFile);
    return tempFile;
    // 임시 파일 삭제는 FileDownloadStreamer의 finally 블록에서 처리
}
결과
다운로드 중 이탈 후 재시도해도 이전 잔여 파일의 영향을 받지 않아 파일 손상이 완전히 사라졌습니다. UUID 기반 난수 파일명은 업로드 저장 시 원본 파일명 충돌 방지에도 동일하게 적용된 일관된 설계 원칙입니다.
